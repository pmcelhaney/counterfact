# ADR xxx: Multiple API Support

## Status

Draft

## Context

Counterfact currently supports exactly one OpenAPI specification per server instance. Teams working with micro-service architectures or versioned APIs need to mock several specs simultaneously under a single server and a single generated code tree.

The key design questions are:

1. **How does the user declare multiple specs?** The CLI, a config file, or directory conventions are the main candidates.
2. **How is the route prefix for each API determined?** Users should be able to supply an explicit prefix; the system should have a sensible fallback chain for when they don't.
3. **How is generated code laid out?** All APIs must share a single destination directory so that a root `_.context.ts` can hold state shared across all of them, while individual API types must not clash.
4. **What happens to REPL, Swagger UI, watch mode, and backwards compatibility?**

Three designs have been drafted. No decision has been taken yet.

## Decision

_Not yet taken._

## Options

### Option A: Repeated `--spec` CLI flag with embedded prefix

Extend the existing `--spec` flag to accept an optional prefix using a `+` separator, and allow it to be repeated:

```
--spec [optional-prefix+]openapi-spec-location
```

All APIs share a single destination directory (the positional `[destination]` argument). Generated routes land in a subdirectory per prefix slug.

**Example invocation**

```sh
counterfact ./api \
  --spec /users+./specs/users.yaml \
  --spec /products+./specs/products.yaml \
  --port 3100 --serve --generate
```

When the prefix is omitted, it is inferred through the fallback chain (see §Root path determination below). Today's single-spec positional usage is unchanged:

```sh
counterfact openapi.yaml ./api
```

**Generated code layout**

```
api/                        ← [destination] argument
  _.context.ts              ← shared root context (all APIs)
  users/                    ← prefix /users
    _.context.ts
    routes/
      GET.ts
      {userId}/GET.ts
    .cache/
  products/                 ← prefix /products
    _.context.ts
    routes/
      GET.ts
    .cache/
```

**Pros**

- Minimal change to the existing CLI surface.
- No new files required.
- Explicit, auditable — everything is visible in one shell command.
- Shared `_.context.ts` is a first-class feature of the layout.

**Cons**

- Long commands become unwieldy with many specs.
- `prefix+path` syntax is unusual and may surprise users.
- Not easily repeatable / version-controllable without a wrapper script.

### Option B: Typed `counterfact.config.ts` file

When Counterfact is launched without positional arguments (or with `--config`), it looks for `counterfact.config.ts` (`.js` or `.json`) in the working directory. The file exports a configuration object via a `defineConfig` helper (mirroring the pattern used by Vite and Jest).

**Example config file**

```ts
// counterfact.config.ts
import { defineConfig } from "counterfact";

export default defineConfig({
  port: 3100,
  apis: [
    { name: "Users",    spec: "./specs/users.yaml",    prefix: "/users"    },
    { name: "Products", spec: "./specs/products.yaml", prefix: "/products" },
  ],
  serve: true,
  generate: true,
  watch: true,
});
```

CLI flags override config-file values, enabling ad-hoc overrides without editing the file.

**Generated code layout**

All APIs share a single destination directory and follow the same sibling layout as Option A, with a root `_.context.ts` shared by all route handlers.

**Pros**

- Familiar pattern for TypeScript developers (Vite, Vitest, Jest all use `defineConfig`).
- Fully typed with IDE autocompletion.
- Version-controllable; easy to diff and review.
- Scales naturally to many APIs without long shell commands.
- Enables a programmatic API: `import { counterfact, defineConfig } from "counterfact"`.

**Cons**

- Requires introducing a new file convention.
- A `defineConfig` export must be added to the public package API.
- Slightly higher startup overhead (config file must be parsed/transpiled before the server starts).

---

### Option C: Convention-over-configuration workspace directory

The user places all their OpenAPI documents under a single **workspace directory**. Counterfact scans that directory, pairs each spec with a generated-code subdirectory, and mounts all APIs on a single server. An optional `counterfact.workspace.yaml` sidecar provides overrides without requiring a full config file.

**Example workspace layout**

```
workspace/
  users/
    openapi.yaml          ← spec auto-discovered
    routes/               ← generated by Counterfact
      GET.ts
      {userId}/GET.ts
  products/
    openapi.yaml
    routes/
      GET.ts
  counterfact.workspace.yaml   ← optional overrides
```

**Invocation**

```sh
counterfact --workspace ./workspace --port 3100 --serve --generate
```

**Sidecar (optional overrides)**

```yaml
apis:
  users:
    prefix: /api/v1/users
  products:
    spec: https://example.com/products/openapi.json
    prefix: /api/v1/products
```

**Pros**

- Zero-config for the common case — just drop an `openapi.yaml` in a subdirectory.
- Self-documenting layout: the directory tree mirrors the API surface.
- Supports adding/removing APIs at runtime when `--watch` is active.
- Naturally handles API versioning via nested subdirectories.

**Cons**

- Requires a dedicated workspace directory, which may not suit all project structures.
- Auto-discovery logic adds complexity (glob patterns, naming conventions).
- Shared `_.context.ts` would need to live at the workspace root, somewhat separate from the per-API subdirectories.
- Less explicit than Options A and B: it may not be obvious which files Counterfact is watching.

## Root path determination (all options)

Regardless of which option is selected, the prefix for each API is resolved in this order:

| Priority | Source |
|----------|--------|
| 1 (explicit) | User-supplied prefix (flag, config field, or sidecar) |
| 2 (OpenAPI 2.x) | `basePath` field in the spec |
| 3 (OpenAPI 3.x) | Path component of the first `servers[].url` entry |
| 4 (fallback) | Slugified identifier derived from the spec path or directory name |

If two or more specs resolve to the same root path, Counterfact prints a descriptive error and exits before starting the server.

## Consequences

_To be filled in once a decision is taken._

Whichever option is chosen, the following cross-cutting concerns must be addressed:

- **Backwards compatibility.** Single-spec usage must continue to work exactly as today.
- **Swagger UI.** Each API is served at `/counterfact/swagger/<slug>`. The dashboard lists all mounted APIs.
- **REPL.** A `:api <name>` command switches the active API; tab completion is scoped to the selected API.
- **Watch mode.** A change to one spec re-generates only that API's routes.
- **Dispatcher.** One `Dispatcher` instance is created per spec; the Koa middleware routes requests to the correct dispatcher by prefix.
- **Admin API.** `/_counterfact/api/config` returns an array of API descriptors.

## Advice

- **Consider Option A first** if the primary goal is minimising the number of new concepts introduced. The `prefix+path` syntax is unusual but keeps everything in one place. (Copilot/Claude)
- **Consider Option B** if version-control friendliness and IDE support for configuration are priorities. The `defineConfig` pattern is well understood by the TypeScript community. (Copilot/Claude)
- **Consider Option C** if zero-config developer experience and runtime hot-add/remove of APIs are key requirements. (Copilot/Claude)
- **Options A and B are not mutually exclusive** — the config file (B) could be introduced first, and the repeated `--spec` flag (A) added later as a convenience shorthand. (Copilot/Claude)
- **The shared `_.context.ts` requirement** (all API route handlers must be able to import a common context file) most naturally favours the sibling layout in Options A and B over the co-located layout described in the original Option C draft. (Copilot/Claude)
- **Seek advice on this ADR before beginning implementation**, especially regarding the `_.context.ts` sharing mechanism and whether `defineConfig` should be added to the public API surface. (@pmcelhaney)
